## Проблема
В гостевой ленте пользователь сейчас может публиковать только текст, хотя серверная часть уже умеет принимать и сохранять изображения для `POST /api/feed/posts`. Из-за этого есть разрыв между возможностями API и текущим publish flow в UI.

## Цель
Добавить в гостевую ленту прикрепление фото при публикации поста, не меняя и не ломая текущую role-based navigation модель (`guest` / `passenger` / `driver`) и существующие pre-check шаги публикации.

## Контекст
Issue #182 — source of truth для задачи.

Согласно текущему коду backend уже поддерживает все основные пути передачи изображения:
- `image_base64`
- `image_mime_type`
- `image_url`
- `multipart/form-data`

Также сервер уже умеет:
- валидировать входные данные изображения,
- сохранять изображение в `/uploads/feed/...`,
- отдавать данные так, чтобы фронт мог отрисовать медиа в карточке поста.

Пробел находится в frontend publish flow: в форме публикации отсутствует UI выбора файла и отправка изображения в запросе создания поста.

## Scope
- Добавить в `public/guest_feed.html` UI для выбора/очистки одного изображения в форме создания поста.
- Расширить `public/web/js/feed.js` так, чтобы `addNewPost()` мог отправлять изображение вместе с текстом.
- Поддержать клиентскую валидацию типа и размера файла до отправки.
- Сохранить текущий сценарий публикации без фото без изменений.
- Добавить понятные сообщения об ошибках загрузки/валидации.
- Проверить корректное отображение фото в ленте после публикации и после перезагрузки.
- Обновить документацию и тесты под фактический контракт.

## Не входит в scope
- Редактор/обрезка/фильтры изображений.
- Множественная загрузка файлов.
- Drag-and-drop интерфейс.
- Видео-вложения.
- Внешние CDN/облачные хранилища.
- Редизайн role-based navigation или профилей.

## Предлагаемая реализация
1. **Frontend (минимально безопасный инкремент):**
   - добавить UI выбора файла рядом с полем текста;
   - хранить выбранный файл в состоянии формы;
   - при публикации использовать JSON-путь (`image_base64` + `image_mime_type`) как первый шаг, т.к. он уже поддержан backend;
   - очищать состояние файла после успешной публикации.

2. **Backend:**
   - без больших изменений; использовать уже существующую поддержку `image_base64` / `image_mime_type` / `image_url` / `multipart/form-data`;
   - при необходимости внести только точечные правки сериализации/валидации, не меняя общий контракт.

3. **UX и стабильность flow:**
   - сохранить все текущие pre-check и ограничения публикации;
   - не менять существующую role-based navigation модель;
   - добавить понятные ошибки для неподдерживаемого типа файла и превышения лимита.

## Acceptance Criteria
- Пользователь может прикрепить одно фото к посту в гостевой ленте.
- Публикация с фото проходит успешно через `POST /api/feed/posts`.
- Публикация без фото продолжает работать как раньше.
- Фото отображается в карточке поста сразу после публикации и после перезагрузки страницы.
- Ошибки валидации (формат/размер) показываются пользователю понятным текстом.
- Не нарушены текущие pre-check и role-based переходы (`guest` / `passenger` / `driver`).
- Документация и тесты синхронизированы с фактическим поведением.

## Технические заметки
Backend-опоры, которые уже есть и должны использоваться как baseline:
- `validate_image_url_metadata`
- `extract_image_from_json_payload`
- `image_bytes_to_stored_url`
- `parse_multipart_form_data`

Дополнительно важно сохранить backend-ограничения по размеру и допустимым MIME-типам, чтобы frontend-валидация не расходилась с сервером.

## Риски
- Несовпадение лимитов и MIME-ограничений между frontend и backend.
- Повышенная нагрузка на память на мобильных при base64 для крупных изображений.
- Регресс существующего publish flow без изображения.
- Конфликт полей при смешении разных форматов (`media[]`, `image_url`, `image_base64`).
- Неполное тестовое покрытие multipart-пути при наличии серверной поддержки.

## Тестирование
- Успешная публикация поста с фото.
- Успешная публикация поста без фото.
- Ошибка для неподдерживаемого MIME-типа.
- Ошибка для слишком большого файла.
- Корректная отрисовка изображения в ленте (сразу и после reload).
- Проверка отсутствия регрессии в pre-check и role-based навигации.

# Epic: Mobile UI polish for Driver Profile, Documents, and Publish Flow

**Repo:** `MrZelus/BazarDrive`  
**Area:** `web / mobile / profile / documents / publish UX`

## Цель

Привести мобильный интерфейс BazarDrive к более цельному, спокойному и понятному сценарию для водителя:

- уменьшить визуальный шум
- усилить иерархию экранов
- убрать дубли действий
- сделать статусы человеческими
- развести разные сценарии: профиль, документы, публикация, заказ

---

## Проблемы по текущим экранам

### 1. Слабая визуальная иерархия

Сейчас многие карточки выглядят одинаково важными:

- профиль
- документы
- статусы
- требования
- публикационный профиль
- CTA-кнопки

Из-за этого экран читается тяжело, особенно на мобильном.

### 2. Дублирование CTA

На экранах документов повторяется `Добавить документ`, что создаёт лишнюю конкуренцию действий.

### 3. Технические сообщения видны пользователю

В UI попадают:

- `Failed to fetch`
- `Загрузка...`
- `пусто`

Это ломает ощущение готового продукта.

### 4. Смешение сценариев

На разных экранах рядом живут:

- публикация поста
- поиск по ленте
- подтверждение заказа
- документы водителя

Это разные пользовательские потоки, их нужно визуально и навигационно развести.

### 5. Перегруженные текстовые блоки

Юридические требования и правила показаны длинными текстовыми карточками, которые на мобильном читаются тяжело.

### 6. Роль/сегмент водителя отображается неочевидно

В некоторых состояниях верхний сегмент выглядит пустым или визуально недозаполненным.

---

## Expected UX outcome

После доработки интерфейс должен:

- быстрее объяснять текущее состояние профиля
- явно подсказывать следующий шаг
- не дублировать действия
- не показывать технические ошибки как пользовательские состояния
- быть удобным для работы с документами одной рукой с телефона
- выглядеть как единая mobile-first система, а не как набор отдельных блоков

---

## Scope

### A. Hero-блок профиля

- [ ] собрать верх профиля в один главный summary-block
- [ ] показать:
  - [ ] роль
  - [ ] форма деятельности
  - [ ] налоговый режим
  - [ ] главный статус допуска
  - [ ] прогресс заполнения
- [ ] оставить один главный CTA:
  - [ ] `Продолжить заполнение`
  - или `Добавить документы`
- [ ] убрать визуальную фрагментацию из нескольких равнозначных мини-карточек

**Пример состояния:**

- `Профиль водителя`
- `Заполнено 4/11`
- `Не готов к заказам`
- CTA: `Продолжить заполнение`

### B. Статусы и ошибки

- [ ] убрать из пользовательского UI технические строки:
  - [ ] `Failed to fetch`
  - [ ] `пусто`
  - [ ] сырые loading-строки там, где можно показать skeleton
- [ ] ввести человеко-понятные состояния:
  - [ ] `Не удалось загрузить данные`
  - [ ] `Профиль ещё не заполнен`
  - [ ] `Документы ещё не добавлены`
- [ ] унифицировать chip/badge статусы по смыслу:
  - [ ] loading
  - [ ] empty
  - [ ] warning
  - [ ] success
  - [ ] review/pending

### C. Навигация и табы

- [ ] пересобрать иерархию:
  - [ ] сначала выбор роли
  - [ ] потом role-specific вкладки
- [ ] исключить ситуацию, когда доступны неочевидные/пустые табы
- [ ] убедиться, что driver-only разделы не рендерятся полу-пустыми
- [ ] сделать активный tab заметнее остальных

### D. Документы водителя

- [ ] разделить документы по состояниям:
  - [ ] обязательные
  - [ ] на проверке
  - [ ] подтверждённые
  - [ ] история / архив
- [ ] всегда показывать активный открытый путевой лист первым
- [ ] визуально выделить открытый путевой лист как текущий рабочий документ
- [ ] в карточках документов оставить только локальные действия:
  - [ ] `Открыть`
  - [ ] `Удалить`
  - [ ] `Закрыть путевой лист`
- [ ] убрать повторяющиеся глобальные CTA из середины контента

### E. Sticky action bar

- [ ] оставить один sticky primary CTA внизу:
  - [ ] `Добавить документ`
- [ ] вторичные действия перенести в overflow / `Ещё`
- [ ] проверить, чтобы sticky bar не перекрывал контент и последние карточки
- [ ] добавить достаточный bottom padding для scroll area

### F. Публикационный профиль

- [ ] оформить блок публикационного профиля как отдельный спокойный section
- [ ] уточнить иерархию полей:
  - [ ] отображаемое имя
  - [ ] email
  - [ ] телефон
  - [ ] о себе
- [ ] добавить inline validation:
  - [ ] email
  - [ ] телефон
- [ ] optional поля визуально отделить от обязательных
- [ ] кнопку `Сохранить профиль публикации` делать:
  - [ ] primary только при изменениях
  - [ ] disabled / secondary в сохранённом состоянии

### G. Юридические и справочные блоки

- [ ] длинные требования свернуть в accordion / collapsible sections
- [ ] вынести ключевое summary наверх:
  - [ ] что обязательно
  - [ ] что мешает допуску
  - [ ] что нужно сделать дальше
- [ ] полный текст правил показывать по раскрытию, а не сразу простынёй

### H. Экран ленты и заказа

- [ ] развести в разные сценарии:
  - [ ] создание поста
  - [ ] поиск по ленте
  - [ ] подтверждение заказа
- [ ] не держать на одном экране блоки из разных сценариев без сильной причины
- [ ] проверить mobile flow для bottom nav:
  - [ ] `Лента`
  - [ ] `Правила`
  - [ ] `Профиль`

## UI acceptance criteria

- [ ] первый экран профиля читается за 3–5 секунд без прокрутки
- [ ] пользователь понимает:
  - [ ] готов ли он к заказам
  - [ ] сколько шагов осталось
  - [ ] что делать дальше
- [ ] на экране документов только один глобальный primary CTA
- [ ] нет сырых технических ошибок в UI
- [ ] открытый путевой лист визуально отличается от закрытых
- [ ] нижняя навигация и sticky CTA не конфликтуют
- [ ] длинные информационные блоки не перегружают mobile viewport

## Design notes

Визуально стоит держать курс на:

- light mobile-first UI
- чистые карточки
- меньше равнозначных блоков
- сильный один CTA
- более “Instagram-inspired rhythm”, но без декоративного шума

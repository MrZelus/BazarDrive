# #172 — Documents and Trust Layer: план декомпозиции и автоматизации

Документ фиксирует рабочую структуру эпика и минимальный набор автоматизации, чтобы изменения были непрерывно проверяемыми.

## Проект и подзадачи

Рекомендуемая декомпозиция эпика #172:

- [x] **172.1** Общий блок профиля публикации доступен всем ролям (`driver`, `passenger`, `guest`).
- [x] **172.2** Driver-only блок документов виден только роли `driver`.
- [x] **172.3** Смена ролей не оставляет недоступные элементы активными.
- [x] **172.4** Статусы документов (`uploaded`, `checking`, `approved`, `rejected`, `expired`) отображаются понятными метками.
- [x] **172.5** Обработаны состояния документов `empty`, `loading`, `error`.
- [x] **172.6** Документация, HTML/JS и тесты синхронизированы.
- [x] **172.7** UI/API готовы к будущим trust badges и verification signals.

### Статусы в GitHub Project (built-in automation)

- Новые задачи и PR автоматически получают статус **Todo**.
- При переводе задачи в работу — **In Progress**.
- После merge/close — **Done**.

## CI-проверки для #172

Workflow: `.github/workflows/ci-documents.yml`

Запускается:
- на `pull_request` и `push` в `main`;
- вручную (`workflow_dispatch`);
- еженедельно по расписанию (cron) для регулярной проверки.

Что проверяем в CI:

1. Контракт синхронизации документов между schema/types/html.
2. Smoke UI-проверки вкладки документов.
3. Флоу публикации с обязательностью профиля.
4. Синхронизацию документации и acceptance-критериев #172.

## Автообновление Project

Для PR, связанных с #172:

Реализуется отдельным workflow на события `pull_request` (`opened`, `ready_for_review`, `closed`) через GitHub GraphQL API.

## Зависимости между issues (blocked / blocking)

Для подзадач 172.1–172.7 фиксируем блокировки через встроенные связи GitHub Issues:

1. Открыть issue, для которой нужна зависимость.
2. На правой панели выбрать **«Связи»**.
3. В выпадающем списке выбрать тип связи:
   - **«Пометить как заблокированный»** — если текущая задача зависит от другой;
   - **«Пометить как блокировку»** — если текущая задача блокирует другую.
4. В диалоге выбрать связанные issues и сохранить.

Минимальное правило для #172: если подпункт нельзя завершить из-за другого подпункта или внешней задачи, связь `blocked by`/`blocking` должна быть создана сразу, чтобы узкие места были видны в Project и списке issues.

## Еженедельный отчёт

Еженедельный запуск CI (cron) используется как контрольный отчёт по регрессиям.

Рекомендуемый формат отчёта в issue #172:

- какие проверки пройдены;
- какие проверки упали;
- список блокирующих пунктов по подзадачам 172.1–172.7.

## Definition of Done для веток по #172

Перед merge проверяем:

- [ ] acceptance-критерии в коде и UI выполнены;
- [ ] тесты из `ci-documents.yml` проходят;
- [ ] docs обновлены вместе с кодом;
- [ ] PR привязан к #172 и корректно отражается в Project.

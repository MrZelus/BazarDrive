# План реализации issue #160: навигация по профилю по ролям (driver/passenger/guest)

## Краткая сводка
Профиль пользователя должен показывать только разрешённые для его роли вкладки: `overview`, `documents`, `security`, а для `driver` дополнительно `taxi_ip` и `payouts`.

Сейчас `passenger`/`guest` видят все вкладки, а переключатель ролей не ограничивает доступ. Требуется:
- ввести жёсткую матрицу `роль → доступные вкладки`;
- скрыть `Такси / ИП` и `Выплаты` для non-driver;
- программно запрещать недоступные вкладки с fallback на `overview`;
- разделить `Документы` на общий блок публикации профиля (для всех) и блок документов водителя (только driver);
- перенести индикатор готовности профиля публикации (`guestProfileStatus`) из `Обзора/Безопасности` в `Документы`;
- изменить роль по умолчанию на `guest`;
- обновить документацию и тесты под новую матрицу.

## Out of scope
- Полная переделка дизайна профиля.
- Изменения backend/API.
- Кардинальная перестройка архитектуры табов вне role-based доступа.

## Затронутые участки кода
- `guest_feed.html` — профильные кнопки и экраны (`profile-tab-*`).
- `web/js/feed.js` — `setActiveProfileTab`, `setRole`, `initialRole`, логика видимости блоков документов.
- `docs/feed_navigation_publish_flow.md`, `docs/driver_profile_wireframe_spec.md`.
- Тесты навигации/ролей (`tests/test_main_tabs_a11y.py`, `tests/test_feed_publish_profile_navigation_flow.py` и смежные).

## Зависимости и риски
- Риск сломать forced publish flow (`Feed -> Profile -> Documents`) после перестройки вкладки `Документы`.
- Риск “обхода” через прямой JS tab-switch, если ограничить только видимость кнопок.
- Риск UI-мерцаний и неконсистентного состояния ARIA (`aria-hidden`, `aria-selected`, `disabled`).
- Изменение default role на `guest` может повлиять на начальные пользовательские сценарии.

## Поэтапный план и подзадачи

### 1) Ввести mapping `role -> allowed tabs` (P0)
**Цель:** централизовать матрицу доступа:

```js
const ROLE_TABS = {
  driver: ['overview', 'taxi_ip', 'documents', 'payouts', 'security'],
  passenger: ['overview', 'documents', 'security'],
  guest: ['overview', 'documents', 'security']
};
```

**Файлы:** `web/js/feed.js`.

**Критерии приёмки:**
- При инициализации и при смене роли UI скрывает/деактивирует недоступные табы.
- Для `driver` доступны все 5 вкладок.

**Проверка:**
- Переключить на `passenger`/`guest`: `taxi_ip` и `payouts` не отображаются и недоступны по ARIA.

**Риски:**
- Ошибка проверки роли может скрыть все вкладки; контрольный инвариант — `overview` всегда доступен.

---

### 2) Запретить программный переход на недоступные табы (P0)
**Цель:** доработать `setActiveProfileTab(tab)` так, чтобы недоступная вкладка заменялась на `overview`.

**Файлы:** `web/js/feed.js`.

**Критерии приёмки:**
- В `guest/passenger` вызовы `setActiveProfileTab('taxi_ip')` и `setActiveProfileTab('payouts')` не открывают эти табы.
- Активной остаётся/становится `overview`.

**Проверка:**
- Прямой вызов из консоли на запрещённый таб в не-driver роли.

**Риски:**
- Рекурсивный цикл fallback; нужен безрекурсивный guard.

---

### 3) Авто-fallback при смене роли (P0)
**Цель:** в `setRole` проверять текущий активный таб и при необходимости переводить в `overview`.

**Файлы:** `web/js/feed.js`.

**Критерии приёмки:**
- На `payouts`/`taxi_ip` смена роли на `guest/passenger` мгновенно возвращает в `overview`.

**Проверка:**
- Сценарий: `driver -> payouts -> guest`.

**Риски:**
- Потеря несохранённых данных; важно как минимум сохранить предсказуемое поведение и уведомление (если предусмотрено).

---

### 4) Default role = `guest` (P0)
**Цель:** изменить fallback инициализации роли на `guest`.

**Файлы:** `web/js/feed.js`.

**Критерии приёмки:**
- После очистки `localStorage` приложение стартует в роли `guest`.

**Проверка:**
- Проверить значение `bazardrive_selected_role` после первой инициализации.

**Риски:**
- Возможные побочные эффекты для существующих сценариев, ожидающих `driver` по умолчанию.

---

### 5) Разделить контент вкладки `Документы` и перенести статус готовности (P1)
**Цель:**
- В `Документы` держать:
  1) блок публикации профиля (для всех ролей),
  2) блок документов водителя (только `driver`).
- Перенести `guestProfileStatus` в `Документы`.

**Файлы:** `guest_feed.html`, `web/js/feed.js`.

**Критерии приёмки:**
- `guest/passenger`: виден только publish-profile блок.
- `driver`: видны оба блока.
- Статус готовности отображается в `Документы` и обновляется корректно.

**Проверка:**
- Ручной smoke для всех ролей + forced publish flow (`Feed -> Profile -> Documents`).

**Риски:**
- Дублирование текста/копирайта и регресс публикации при переносе статуса.

---

### 6) Обновить docs и тесты под новую матрицу (P1/P2)
**Цель:** синхронизировать документацию, навигационные и ролевые тесты.

**Файлы:**
- `docs/feed_navigation_publish_flow.md`
- `docs/driver_profile_wireframe_spec.md`
- `tests/*role*`, `tests/*navigation*`, связанные smoke/regression тесты

**Критерии приёмки:**
- Документация соответствует новой матрице вкладок.
- Тесты покрывают:
  - видимость табов по ролям,
  - запрет прямого tab-switch,
  - fallback при смене роли,
  - сохранение forced-flow публикации.

**Проверка:**
- Прогон CI-пакета профильной навигации и полного релевантного pytest-набора.

**Риски:**
- Хрупкие тесты из-за привязки к верстке; лучше опираться на `id`/`aria-*`.

## Приоритеты
- **P0:** матрица ролей + ограничения переходов + fallback + default guest.
- **P1:** перестройка `Документы` + перенос статуса готовности профиля публикации.
- **P2:** полная синхронизация docs/тестов и стабилизация CI.

## Рекомендуемый первый шаг
Сначала внедрить `ROLE_TABS` в `feed.js` и применить её на инициализации/смене роли/переключении вкладок. Это базовый слой, на который безопасно опираются остальные изменения.

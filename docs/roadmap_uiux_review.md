# UI/UX аудит roadmap для BazarDrive (лента + такси)

Дата: 2026-04-03
Роль: Product Design review (mobile aggregator)
Исходный документ: `docs/roadmap.md`

## 1) UI-риски (узкие места интерфейса)

1. **Риск перегруженного первого экрана (Feed + Taxi entrypoint)**
   - В roadmap есть акцент на risk domains и backend/workstreams, но нет явного паттерна главного экрана: что является primary action в первый визит (скролл ленты vs «Заказать такси»).
   - Без приоритезации визуальной иерархии возможна конкуренция CTA и падение конверсии в обе оси.

2. **Риск “невидимой” стоимости ошибок в критических статусах поездки**
   - Описан taxi lifecycle как направление, но не определён UX-набор обязательных состояний и recovery-переходов на уровне экранов.
   - В реальности это бьёт по поддержке: пользователь не понимает, что происходит на этапах «поиск машины», «водитель ждёт», «поездка завершена».

3. **Риск разрыва feed→taxi конверсии**
   - Есть mention deep-link/prefill/attribution, но нет UX-ограничений на количество шагов и нет target на “tap-to-order” сценарий.
   - Это создаёт «бутылочное горлышко» в рекламных карточках: пользователь видит оффер, но теряется в промежуточных экранах.

4. **Риск позднего обнаружения cash-only фрустрации**
   - В roadmap payment safety покрыт технически, но нет продуктового требования “cash-only disclosure first”.
   - Если это не акцентировать на ранних шагах, пользователи узнают про наличные слишком поздно (на финише).

5. **Риск accessibility debt на мобильном рантайме**
   - Упомянуты interruptions/resilience, но accessibility не вынесена в acceptance criteria (контраст, tappable зоны, динамический текст).
   - Для сценариев «на ходу» это критично: особенно карта, CTA заказа, звонок водителю, отмена.

## 2) Пропущенные экраны / макеты, которые стоит добавить в roadmap

### Лента
- Loading skeleton для карточек и шапки.
- Пустая лента (первый запуск / нет релевантного контента).
- Ошибка сети + retry + offline hint.
- Конец списка (end-of-feed state).
- Состояние “обновляем” (pull-to-refresh).

### Такси
- Экран выбора точки на карте: `pin drag`, геокодинг, подтверждение pickup.
- Экран «поиск машины» с прогрессом + понятной отменой.
- Экран «водитель ждёт» с таймером, платой за ожидание (если применимо) и быстрыми действиями.
- Экран завершения поездки: итог, сумма, **оплата наличными** (подтверждение факта оплаты), issue/report CTA.
- Экран потери сети во время активного заказа + восстановление состояния после reconnect.

### Профиль / IA
- Раздельные вкладки/секции: Личные данные / История поездок / Настройки ленты.
- Экран «Платёжные настройки» в виде read-only cash policy (без ложных affordances под карту).
- Экран управления уведомлениями (поездки vs лента).

### Cross-domain (лента → такси)
- Deep-link landing из карточки акции с предзаполненным destination/promo.
- Промежуточный экран проверки параметров заказа (pickup, destination, cash-only, promo).
- Экран “promo applied / promo invalid” без потери контекста заказа.

## 3) UX-рекомендации (3–5 идей для бесшовности)

1. **Сделать единый Home с одним первичным CTA**
   - На главном экране один доминирующий action («Заказать такси»), лента — вторичная, но заметная.
   - Снизит когнитивную конкуренцию двух продуктовых осей.

2. **Ограничить feed→order путь до 2–3 тапов максимум**
   - Тап по карточке → prefilled order sheet → подтверждение.
   - Убрать лишние промежуточные экраны через bottom-sheet и prefill.

3. **Показывать cash-only policy в трёх точках**
   - На карточке/оффере (бейдж), в экране подтверждения заказа, на экране завершения поездки.
   - Это уменьшит фрустрацию и количество конфликтов «почему нельзя картой».

4. **Ввести state matrix как обязательный артефакт roadmap**
   - Для каждого ключевого экрана зафиксировать: loading / empty / error / success / offline / interrupted recovery.
   - Синхронизировать с QA acceptance и тест-кейсами.

5. **Добавить accessibility guardrails в DoD**
   - Минимальные touch targets, контраст текста/кнопок, поддержка системного масштабирования шрифта.
   - Критично для использования «на ходу» и для снижения ошибок на карте.

## 4) Проверка по заданным критериям

- **Консистентность:** концептуально отражена (hybrid product), но визуальная иерархия главного экрана не зафиксирована.
- **States:** частично покрыто на уровне направлений; конкретные UI-состояния ленты/поездки нужно явно добавить.
- **IA профиля:** доменно логично, но отсутствует явное UX-разделение пользовательских разделов.
- **User Flow feed→taxi:** отмечен как риск-домен, но не определены KPI по числу шагов/дропа.
- **Accessibility:** в текущем тексте roadmap не выделена отдельным нефункциональным требованием.
- **Cash-only:** технический payment focus есть, но UX-коммуникация cash-only недостаточно формализована.

## 5) Что добавить в roadmap как acceptance criteria

- UX state matrix для ключевых экранов ленты и поездки.
- Максимум шагов из промо-карточки в подтверждение заказа (SLA по кликам).
- Обязательная видимость cash-only policy на 3 этапах пути.
- Accessibility baseline (контраст, размер зон нажатия, шрифтовые масштабы).
- Profile IA contract (структура разделов и навигации).

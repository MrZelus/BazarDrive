# Сценарий входа и онбординга водителя в BazarDrive

## 1. Цель документа

Этот документ описывает целевой сценарий входа водителя в BazarDrive:
- первый вход;
- повторный вход;
- выбор роли водителя;
- создание и заполнение профиля;
- загрузка обязательных документов;
- прохождение проверки;
- допуск к выходу на линию;
- блокирующие условия и edge cases.

Документ нужен как единая точка синхронизации между:
- frontend UI;
- backend/API;
- Telegram bot;
- документацией и QA.

---

## 2. Основные сущности

### 2.1. Пользователь
Пользователь системы, который может иметь одну или несколько ролей.

### 2.2. Роль `driver`
Роль, дающая доступ к водительскому профилю, документам, выходу на линию и принятию заказов.

### 2.3. Профиль водителя
Расширенный профиль с персональными данными, данными автомобиля, статусами допуска и связанными документами.

### 2.4. Документы водителя
Обязательные и дополнительные документы, влияющие на возможность работы.

### 2.5. Верификация
Процесс проверки профиля и документов, после которого водитель либо допускается к работе, либо получает отказ/замечания.

---

## 3. Сценарии верхнего уровня

### 3.1. Первый вход водителя
1. Пользователь открывает приложение / бот / web-интерфейс.
2. Проходит авторизацию или идентификацию.
3. Выбирает роль `driver`, если роль ещё не закреплена.
4. Создаёт профиль водителя.
5. Заполняет обязательные данные.
6. Загружает обязательные документы.
7. Отправляет профиль на проверку.
8. После одобрения получает доступ к выходу на линию.

### 3.2. Повторный вход водителя
1. Пользователь повторно открывает систему.
2. Система восстанавливает сессию или предлагает повторную авторизацию.
3. Пользователь попадает в экран профиля/статуса водителя.
4. Система показывает актуальные блокеры:
   - нет обязательного документа;
   - документ истёк;
   - профиль на проверке;
   - профиль отклонён;
   - можно выходить на линию.

### 3.3. Вход водителя с неполным профилем
1. Пользователь входит в систему.
2. Система определяет, что профиль водителя заполнен не полностью.
3. Пользователь направляется в сценарий дозаполнения профиля.
4. До завершения обязательных шагов выход на линию запрещён.

---

## 4. Пошаговый flow первого входа

### Шаг 1. Открытие системы
Точка входа:
- web;
- Telegram bot;
- мобильный web/app shell.

Ожидаемый результат:
- пользователь попадает на экран входа / идентификации / выбора сценария.

### Шаг 2. Авторизация / идентификация
Возможные варианты:
- Telegram identity;
- телефон;
- одноразовый код;
- существующая сессия.

Ожидаемый результат:
- система знает, кто пользователь;
- создаётся или находится запись пользователя.

### Шаг 3. Определение роли
Если у пользователя ещё нет активной роли водителя:
- показывается выбор роли;
- пользователь выбирает `driver`.

Если роль уже назначена:
- сценарий выбора роли пропускается.

### Шаг 4. Создание профиля водителя
Пользователь заполняет:
- ФИО;
- телефон и/или email;
- регион работы;
- город / зона работы;
- сведения об автомобиле;
- данные деятельности (ИП / самозанятость / иная модель, если поддерживается).

Минимальный ожидаемый результат:
- создаётся профиль водителя со статусом `draft` / `incomplete` / аналогичным системным состоянием.

### Шаг 5. Проверка заполненности профиля
Система валидирует:
- обязательные поля заполнены;
- формат данных корректный;
- ключевые блоки профиля собраны.

Если есть ошибки:
- пользователь остаётся в сценарии редактирования;
- получает понятные ошибки по полям;
- не может перейти к допуску на линию.

### Шаг 6. Загрузка документов
Пользователь загружает обязательные документы.

Базовый набор может включать:
- водительское удостоверение;
- СТС;
- разрешение такси;
- ИНН / ОГРНИП / связанные регистрационные документы;
- ОСАГО;
- диагностическую карту;
- путевой лист, если он обязателен для рабочего сценария.

### Шаг 7. Проверка completeness документов
Система определяет:
- каких документов не хватает;
- какие документы просрочены;
- какие документы на проверке;
- какие документы отклонены.

Если комплект неполный:
- профиль не допускается к работе;
- пользователь получает список missing docs.

### Шаг 8. Отправка на верификацию
Когда обязательный профиль и документы заполнены:
- профиль получает статус `pending_verification` / `checking` / аналогичный;
- пользователь видит, что данные отправлены на проверку.

### Шаг 9. Решение по верификации
Возможные исходы:
- `approved` — водитель допущен к работе;
- `rejected` — водитель получает причину отказа;
- `needs_fix` / аналогичный статус — требуется корректировка данных.

### Шаг 10. Допуск к выходу на линию
Только после успешной проверки система разрешает:
- включить статус `онлайн`;
- открыть смену;
- получать и принимать заказы.

---

## 5. Повторный вход водителя

При повторном входе система должна быстро определить состояние пользователя и показать один из сценариев:

### 5.1. Водитель полностью готов к работе
Показывается:
- профиль активен;
- документы в порядке;
- нет блокеров;
- можно выходить на линию.

### 5.2. Профиль на проверке
Показывается:
- статус ожидания;
- список уже загруженных данных;
- уведомление, что выход на линию пока недоступен.

### 5.3. Есть missing / expired документы
Показывается:
- конкретный список проблемных документов;
- CTA: загрузить, обновить, исправить.

### 5.4. Профиль отклонён
Показывается:
- причина отказа;
- список необходимых исправлений;
- действие: повторная отправка на проверку.

---

## 6. Блокирующие условия выхода на линию

Выход на линию должен быть запрещён, если:
- не завершено создание профиля водителя;
- не заполнены обязательные поля;
- отсутствуют обязательные документы;
- документы истекли;
- профиль находится в статусе `rejected`;
- профиль находится в статусе `pending_verification`, если бизнес-правила не разрешают ограниченный доступ;
- отсутствует активный путевой лист, если он обязателен для смены;
- есть иные compliance-ограничения.

---

## 7. Основные статусы профиля водителя

Рекомендуемые логические состояния:
- `draft`
- `incomplete`
- `pending_verification`
- `approved`
- `rejected`
- `blocked`

Названия могут адаптироваться к реализации, но смысл должен сохраняться.

---

## 8. Основные статусы документов

Рекомендуемые состояния:
- `uploaded`
- `checking`
- `approved`
- `rejected`
- `expired`
- `missing`
- `expiring_soon`

---

## 9. UI/UX-требования к сценарию входа водителя

Система должна:
- явно показывать текущий статус профиля;
- явно показывать, чего не хватает для допуска к работе;
- не прятать блокеры в глубине интерфейса;
- позволять быстро дозаполнить профиль;
- сохранять введённые данные при сетевых ошибках;
- показывать inline-ошибки и общий статус сценария;
- не разрешать пользователю думать, что он вышел на линию, если допуск ещё не получен.

---

## 10. Edge cases

### 10.1. Пользователь начал регистрацию, но бросил процесс
При следующем входе:
- сценарий продолжается с последнего незавершённого шага;
- уже заполненные данные не теряются.

### 10.2. Часть документов отклонена
- пользователь видит только проблемные документы;
- может заменить их без полного пересоздания профиля.

### 10.3. Документы были валидны, но позже истекли
- выход на линию блокируется до обновления;
- пользователь получает предупреждение заранее, если поддерживается `expiring_soon`.

### 10.4. Пользователь сменил автомобиль
- система должна потребовать обновление связанных документов;
- допуск может быть временно ограничен до повторной проверки.

### 10.5. Путевой лист обязателен только в рабочей смене
- профиль может быть approved;
- но выход в `онлайн` блокируется до открытия путевого листа.

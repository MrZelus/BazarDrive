# Driver Menu Map

## 1. Цель документа

Этот документ описывает **карту меню водителя** в BazarDrive и связывает навигацию между:
- Telegram;
- Web;
- профильными разделами;
- рабочими сценариями;
- переходами к заказу, смене, документам и выплатам.

Документ нужен, чтобы Telegram и Web были разными интерфейсами одной системы, а не двумя несовпадающими UX-ветками.

## 3. Навигационные цели

Навигация водителя должна помогать быстро решать 4 группы задач:
- понять текущий статус допуска и смены;
- исправить блокеры профиля / документов;
- получить доступ к активному заказу;
- перейти к истории, выплатам и настройкам.

## 5. Принципы различия Telegram и Web

### Telegram
Подходит для:
- быстрых действий;
- уведомлений;
- управления активным заказом;
- коротких рабочих сценариев;
- компактного доступа к статусам.

### Web
Подходит для:
- расширенного редактирования;
- управления документами;
- загрузки файлов;
- обзора статусов;
- истории, аналитики и выплат.

При этом логика разделов и статусов должна быть общей.

---

## 6. Основные разделы и их назначение

### 6.1. Профиль
Содержит:
- личные данные;
- рабочие данные;
- статус профиля;
- блокеры допуска.

### 6.2. Авто
Содержит:
- данные автомобиля;
- связанные документы;
- статус пригодности авто для работы.

### 6.3. Документы
Содержит:
- список обязательных и дополнительных документов;
- статусы `uploaded/checking/approved/rejected/expired`;
- CTA на загрузку, замену, повторную отправку.

### 6.4. Путевой лист
Содержит:
- состояние текущего путевого листа;
- загрузку / открытие;
- закрытие после смены;
- предупреждения, если документ обязателен.

### 6.5. Смена
Содержит:
- статус `offline / online / busy / closed` или эквивалент;
- CTA на выход на линию и завершение смены;
- блокеры допуска к открытию смены.

### 6.6. Заказ / Активный заказ
Содержит:
- карточку активного заказа;
- текущий order status;
- CTA для перехода по статусам;
- ветку отмены.

### 6.7. История
Содержит:
- завершённые поездки;
- отменённые заказы;
- при необходимости фильтры и поиск.

### 6.8. Баланс / Выплаты
Содержит:
- начисления;
- выплаты;
- удержания;
- статус выплат.

### 6.9. Поддержка / Настройки
Содержит:
- уведомления;
- язык;
- справку;
- каналы связи с поддержкой.

---

## 7. Обзор как центр системы

Web-раздел `Обзор` должен играть роль дашборда и кратко показывать:
- готовность профиля;
- состояние документов;
- статус путевого листа;
- статус смены;
- наличие активного заказа;
- быстрые CTA в проблемные точки.

Если у пользователя есть блокер, обзор должен вести именно туда, где проблема реально находится.

---

## 8. Правила навигации

Система должна:
- сохранять единый смысл разделов между Telegram и Web;
- не прятать блокирующие разделы;
- давать путь к активному заказу из любой рабочей точки;
- различать редактирование данных и оперативные действия;
- поддерживать прямые переходы по alert / notification / blocker CTA.

---

## 9. Entry points and cross-links

Нужно зафиксировать ключевые переходы:
- из обзора в профиль, документы, смену и активный заказ;
- из блокера документов в нужный тип документа;
- из смены в active order;
- из завершённого заказа в историю;
- из завершения смены в закрытие путевого листа;
- из push / notification сразу в активную сущность.

---

## 10. UI/UX требования

Система должна:
- не допускать конфликтующих названий разделов;
- использовать одинаковые статусы и термины;
- минимизировать число кликов до активного заказа;
- на мобильных экранах удерживать ключевые CTA в зоне быстрого доступа;
- показывать, где находится пользователь и что ещё требует действий.

---

## 11. Edge cases

- пользователь попадает в Web сразу по deep link на документ;
- Telegram ведёт в раздел, которого ещё нет в Web в том же виде;
- есть активный заказ, но водитель открыл историю;
- блокер в профиле неочевиден и должен подсветиться из обзора;
- одновременно есть несколько проблем: документы, путевой лист и rejected verification.

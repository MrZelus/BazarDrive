## 3. Секция 1 — Шапка профиля

**Расположение:** верхняя часть экрана.

**Состав:**
- фото пользователя;
- ФИО;
- роль пользователя;
- форма деятельности;
- налоговый режим;
- общий статус аккаунта.

**Пример содержимого:**
- Фото
- Алексей Смирнов
- Водитель такси
- ИП
- УСН «Доходы»
- Статус: Активен

**Назначение:**
Обеспечить мгновенную идентификацию пользователя и текущего состояния его рабочего профиля.

## 4. Секция 2 — Быстрые статусы

**Расположение:** сразу под шапкой.

**Формат:** горизонтальные карточки/компактные блоки.

**Карточки:**
- Документы: 8/8 подтверждены
- Разрешение такси: активно
- Выплаты: подключены
- ОСАГО: действует до 18.09.2026

**Назначение:**
Дать пользователю обзор критически важных статусов в одном месте.

## 5. Секция 3 — Навигация по вкладкам

**Расположение:** под блоком быстрых статусов.

**Вкладки:**
- Обзор
- Такси / ИП
- Документы
- Выплаты
- Безопасность

**Примечание (role-based):**
- Для `driver` доступна полная 5-вкладочная структура.
- Для `passenger` и `guest` доступны только `Обзор`, `Документы`, `Безопасность`.
- Переходы в `Такси / ИП` и `Выплаты` для non-driver недоступны (включая программные вызовы переключения вкладок).

**Поведение:**
- активная вкладка визуально выделяется;
- переключение выполняется без перезагрузки страницы;
- контент ниже обновляется в зависимости от выбранной вкладки.

### 5.3.1. UX-сценарий кнопки «Добавить документ»

Кнопка **«Добавить документ»** находится в верхней части вкладки **«Документы»** рядом с заголовком списка.

**Состояния кнопки и формы:**
- `idle` — кнопка активна, форма скрыта;
- `loading` — после submit кнопка формы блокируется, показывается статус отправки;
- `success` — документ создан, форма закрывается, список обновляется **без reload страницы**;
- `error` — форма остаётся открытой, ошибки полей показываются рядом с полями, API-ошибка показывается общим alert/toast.

**Edge cases:**
- повторный submit во время `loading` блокируется;
- при сетевой ошибке пользователь может повторить отправку без потери введённых данных;
- если список пустой, после `success` сразу отображается новый документ.

### 5.3.2. Подтверждённый UX-сценарий (для разработки)

Ниже зафиксирован эталонный сценарий, который должен быть реализован в UI/API без отклонений.

1. Пользователь открывает вкладку **«Документы»** и видит кнопку **«Добавить документ»** в состоянии `idle`.
2. По клику на кнопку открывается форма добавления документа:
   - обязательные поля помечены;
   - кнопка отправки активна только если клиентская валидация пройдена.
3. Пользователь нажимает «Сохранить»:
   - форма переходит в `loading`;
   - повторные клики по submit и закрытию формы блокируются до ответа API.
4. Если API возвращает успех:
   - форма переходит в `success` и закрывается;
   - вызывается повторная загрузка списка документов (или optimistic add + сверка);
   - новый документ появляется в списке **без перезагрузки страницы и без смены вкладки**.
5. Если API возвращает ошибку:
   - форма остаётся открытой (`error`);
   - ошибки валидации уровня полей показываются inline рядом с соответствующими полями;
   - ошибки уровня API/сервера (5xx/timeout/network) показываются общим alert/toast;
   - введённые пользователем значения сохраняются (кроме полей, которые сервер требует очистить по безопасности).

**Наблюдаемое поведение после success:**
- список документов обновляется в текущем UI-контексте;
- скролл страницы и активная вкладка не сбрасываются;
- дубли одного и того же документа не должны появляться при медленном ответе API.

**Наблюдаемое поведение после error:**
- пользователь может исправить только невалидные поля и отправить форму повторно;
- при исчезновении сетевой ошибки повторная отправка должна завершаться без полного ресета формы;
- toast/alert не должен перекрывать inline-ошибки полей.

**Критичные edge cases для реализации и тестов:**
- двойной клик по submit;
- закрытие формы по `Esc`/крестику во время `loading` (должно быть заблокировано);
- `409/422` от API (корректная маршрутизация ошибок в поля/общий alert);
- успешное добавление в пустой список (empty-state заменяется карточкой документа);
- повторная попытка после `timeout` без потери заполненных данных.

| Нажатие «Открыть выплаты»       | Переход к финансовому разделу           |
| Переключение 2FA                | Меняется статус двухфакторной защиты    |
| Ошибка API                      | Показывается error-state или banner     |

## 11. Рекомендации по адаптиву

### Мобильный экран

- быстрые статусы лучше показывать в 2 строки карточек;
- вкладки можно сделать горизонтально скроллируемыми;
- нижнюю панель действий закрепить снизу.

### Десктоп

- быстрые статусы в одну линию;
- вкладки под шапкой;
- действия можно вынести вправо или вниз контентного блока.

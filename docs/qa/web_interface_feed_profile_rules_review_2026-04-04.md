# QA-обзор веб-интерфейса: Лента, Профиль, Правила

Дата: 2026-04-04  
Контур: `public/guest_feed.html`, `public/web/js/feed.js`  
Формат: быстрый регрессионный QA-обзор по критическим пользовательским сценариям.

## 1) Область проверки

Проверены три пользовательские зоны:

- **Лента**: поиск, бесконечная прокрутка, единые сообщения ошибок, предзаполнение профиля перед публикацией.
- **Профиль**: обязательность заполнения, сохранение и повторный вход в сценарий публикации.
- **Правила**: наличие ключевых юридических блоков и доступность поиска по документам.

## 2) Что проверено

### Лента (Feed)

- Есть поле поиска и status/live-region для обратной связи пользователю.
- Реализован `IntersectionObserver` и sentinel для lazy loading.
- Есть защита от дублей постов при догрузке.
- При неготовом профиле публикация блокируется с понятным сообщением и редиректом в профиль.

### Профиль (Profile)

- Вкладки профиля и роли ARIA (`tab`, `tabpanel`, `aria-controls`) соответствуют ожидаемому паттерну.
- Сценарий «сохранить профиль и вернуться к публикации» покрыт регрессионным тестом.
- Есть trust/verification индикаторы и сценарий повторной отправки на проверку.

### Правила (Rules)

- Есть отдельный экран правил с поиском и `aria-live` подсказкой статуса.
- В контенте присутствуют карточки с юридическими/комплаенс требованиями для такси.
- Поиск и рендер глубоких секций правил покрыт тестами.

## 3) Результат

Статус по итогам обзора: **PASS (без блокирующих дефектов)**.

- Критические пользовательские сценарии для Ленты / Профиля / Правил проходят.
- Регрессионное покрытие по accessibility, legal-контенту и навигации между экранами актуально.

## 4) Риски и рекомендации

1. **Сохранять стабильность id/aria-атрибутов** в `guest_feed.html` — на них завязаны регрессионные проверки навигации и a11y.
2. **Не размывать единый слой сообщений об ошибках** в `feed.js`; новые ветки ошибок добавлять через существующие словари текстов.
3. Для следующего цикла QA добавить e2e smoke (headless browser) на сценарий:
   - заполнение профиля,
   - публикация поста,
   - поиск по правилам,
   - возврат на ленту и действия с комментариями/реакциями.

## 5) План действий по рекомендациям QA

### Шаг 1. Стабилизировать контракт id/ARIA в UI (Лента / Правила / Профиль)

**Задачи**

- Зафиксировать список критичных селекторов и ARIA-атрибутов (табы, табпанели, поля поиска, status/live-region).
- Добавить отдельный чек «контракт не нарушен» в PR checklist для изменений `public/guest_feed.html`.
- Расширить существующие регрессионные тесты на негативный сценарий (поломка `aria-controls` / `aria-labelledby`).

**Критерий готовности**

- Любая регрессия по id/ARIA приводит к падению тестов до merge.

### Шаг 2. Укрепить единый слой UX-ошибок в `feed.js`

**Задачи**

- Вынести/консолидировать тексты ошибок в единую карту (без точечных строк в бизнес-ветках).
- Добавить регрессионный тест на консистентность текстов для публикации, комментариев и реакций.
- Проверить, что fallback-ошибки и 429/retry-after обрабатываются единообразно.

**Критерий готовности**

- Нет новых `alert`/разрозненных ошибок; все ветки используют общие функции уведомлений и централизованный словарь сообщений.

### Шаг 3. Добавить e2e smoke для критического пользовательского потока

**Сценарий e2e (минимальный)**

1. Открыть `guest_feed.html` и убедиться, что активна вкладка «Лента».
2. Перейти в «Профиль», заполнить обязательные поля и сохранить.
3. Вернуться в «Лента», создать публикацию и проверить успешное отображение.
4. Перейти в «Правила», выполнить поиск по ключевому юридическому термину и проверить выдачу.
5. Вернуться в «Лента», добавить комментарий/реакцию к посту.

**Техническая реализация (рекомендация)**

- Добавить headless smoke-тест в `tests/` как отдельный job nightly/PR-smoke.
- Сохранять артефакты: скриншоты шагов и лог сетевых ошибок.

**Критерий готовности**

- e2e smoke стабильно проходит в CI и формирует артефакты при падении.

### Шаг 4. Порядок внедрения и приоритет

- **P0 (сразу):** Шаг 1 + Шаг 2 (контракт доступности и единые ошибки).
- **P1 (следом):** Шаг 3 (e2e smoke и артефакты).

Ожидаемый эффект: снижение риска регрессий в навигации, доступности и publish-flow при изменениях фронтенда.

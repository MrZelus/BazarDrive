# Driver Profile Components Board

Этот документ фиксирует **единый board ключевых компонентов профиля водителя** в BazarDrive.

Документ нужен как связующая точка между:
- profile UX;
- compliance-блоками;
- сменой;
- активным заказом;
- дальнейшей декомпозицией в UI-kit и frontend-компоненты.

## 2. Что включает board

В board собраны рядом 6 ключевых блоков driver domain:
- обязательные поля;
- документы;
- путевой лист;
- допуск к выходу на линию;
- активный заказ;
- смена.

---

## 3. Компоненты board

### 3.1. Обязательные поля

Назначение:
- показать прогресс заполнения обязательных данных профиля;
- подсветить отсутствующие поля;
- дать быстрый CTA на дозаполнение.

Состав:
- progress indicator;
- filled chips;
- missing chips;
- warning block;
- CTA `Заполнить недостающие поля`.

### 3.2. Документы

Назначение:
- показать статус обязательных документов;
- отделить подтверждённые, проверяемые и проблемные документы;
- вести пользователя к исправлению.

Состав:
- progress / status block;
- chips по состояниям `approved/checking/rejected/expired`;
- warning block;
- CTA `Добавить / заменить документ`.

### 3.3. Путевой лист

Назначение:
- показать состояние путевого листа в контексте смены;
- дать быстрые действия на открытие и закрытие.

Состав:
- trip sheet status;
- shift period;
- warning if missing;
- CTA `Открыть путевой лист` / `Закрыть путевой лист`.

### 3.4. Допуск к выходу на линию

Назначение:
- быть главным gatekeeper-блоком для рабочего режима водителя.

Состав:
- eligibility status: `готов / не готов`;
- checklist по профилю, авто, документам и путевому листу;
- success state;
- blocker state;
- CTA `Выйти на линию` / `Исправить блокеры`.

### 3.5. Активный заказ

Назначение:
- показать текущий order status и дать доступ к следующему действию.

Состав:
- order status;
- route block;
- passenger note;
- CTA `Следующий этап`;
- CTA `Связаться` / `Отменить`.

### 3.6. Смена

Назначение:
- показать текущий shift status и управление доступностью водителя.

Состав:
- shift status;
- readiness checklist;
- CTA `Выйти на линию` / `Завершить смену`;
- recent events / timeline.

## 4. Логические связи между компонентами

Board построен не как набор разрозненных карточек, а как единая система.

### 4.1. Обязательные поля -> Допуск
Незаполненные обязательные поля снижают готовность профиля и влияют на eligibility.

### 4.2. Документы -> Допуск
Проблемные документы блокируют или ограничивают возможность выйти на линию.

### 4.3. Путевой лист -> Допуск
Если путевой лист обязателен, его отсутствие блокирует рабочий режим.

### 4.4. Допуск -> Смена
Только при выполнении обязательных условий водитель может перейти в сценарий открытия смены.

### 4.5. Смена -> Активный заказ
Активный заказ существует в контексте открытой смены.

### 4.6. Активный заказ -> Смена
После завершения или отмены заказа водитель возвращается в контекст смены.

---

## 5. Как использовать board

### Product / UX
Использовать для согласования состава ключевых driver-компонентов.

### Frontend
Использовать как карту декомпозиции на реальные UI-компоненты и состояния.

### Backend / API
Использовать как опору для согласования статусов и блокеров.

### QA
Использовать как основу для сценариев проверки готовности профиля, допуска, смены и order flow.
